# The Difference Between a Software Bug and a Feature Request

**Word Count:** ~600 words
**Target Keywords:** software bug vs feature, QA testing, software development lifecycle

In the high-stress environment of software development, no argument happens more frequently—or causes more tension between clients and development agencies—than the debate over "Bugs" versus "Features."

A client emails the agency: "I found a massive bug! The checkout page doesn't let users save their credit card for later. Fix it immediately!"
The agency replies: "That is not a bug. That is a new feature request. We will have to bill you for an extra 40 hours of development time to build that."

The client feels cheated. The agency feels abused. 

To maintain a healthy outsourcing relationship and keep your project budget under control, you must clearly understand the technical difference between a bug and a feature.

## What is a Bug?
A bug is a defect. It is a failure of the software to behave as it was explicitly designed and documented to behave.

**Examples of a Bug:**
* A user clicks the "Submit" button on a contact form, and the website crashes, displaying a "500 Server Error."
* A customer buys a $10 product, but the shopping cart incorrectly charges them $100.
* The original approved design document explicitly stated the "Cancel" button should be red, but the developer coded it blue.

In all these scenarios, the code is broken or deviates from the agreed-upon requirements. A reputable software agency will fix legitimate bugs for free, as it is a defect in their workmanship.

## What is a Feature Request?
A feature request (often called a "Change Request") is a desire for the software to do something new—something that was never explicitly discussed or documented in the original scope of work. 

**Examples of a Feature Request:**
* "The contact form works perfectly, but we want to add a new dropdown menu so users can select their country." (This requires changing the database schema to store country data).
* "The app works great on iPhones, but can we make it work on Apple Watches too?" (This is an entirely new platform).
* "Users should be able to save their credit card for later." (This requires building complex, PCI-compliant tokenization logic).

Even if a feature seems incredibly "obvious" to the client, if it was not written in the original Jira tickets or requirements document, it is a new feature. Developers cannot read minds; they only build what is written on the blueprint. 

## The Gray Area: "It Doesn't Make Sense"
The most frustrating arguments happen in the gray area. 
The code works exactly as written, but the user experience is terrible. For example, the software technically allows a user to upload a profile picture, but it takes 15 confusing clicks to do it. 
The client calls it a bug ("It's unusable!"). The developer calls it a feature ("It works exactly how you told me to build it!").

## How to Prevent the Argument
The only way to avoid the "Bug vs. Feature" war is rigorous, obsessive documentation before the coding begins. 

This is why you must invest heavily in **UI/UX Prototyping** and **Business Analysts**. If you build a clickable Figma prototype and define every single user story in a Jira ticket, there is no ambiguity. 
If the final software matches the Figma prototype perfectly, any changes are mathematically a Feature Request. If it doesn't match, it is a Bug. 

When partnering with a premium offshore development center, demand a heavy discovery phase. Clarifying the blueprint upfront ensures your budget goes toward building great software, not fighting over definitions.
